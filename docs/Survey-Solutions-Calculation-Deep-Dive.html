2Acquisition Timing Engine

Every acquisition has a completion quarter (e.g., "Q3 26"). The model extracts the year to route financials into the correct annual column, and uses the quarter prefix for synergy phase-in calculations.

Year Extraction

Excel M29 = RIGHT(I29, 2) // "Q3 26" → "26" M30 = RIGHT(I30, 2) // "Q4 26" → "26" M33 = RIGHT(I33, 2) // "Q1 27" → "27"
JavaScript function extractYear(quarterStr) { // quarterStr = "Q3 26" return quarterStr.trim().slice(-2); // "26" } function extractQuarter(quarterStr) { // quarterStr = "Q3 26" return quarterStr.trim().substring(0, 2); // "Q3" }

Year Routing – Placing EBITDA in the Correct FY Column

Each acquisition's EBITDA appears only in the year it completes. This is the fundamental routing mechanism for the entire model.

Excel // Row 48 onwards: "Acquisitions EBITDA by Year of Completion" G48 = IF($M29 = "26", $G29, 0) // FY26 column: show EBITDA if deal completes in '26 H48 = IF($M29 = "27", $G29, 0) // FY27 column: show EBITDA if deal completes in '27 I48 = IF($M29 = "28", $G29, 0) // FY28 column: show EBITDA if deal completes in '28
JavaScript function routeEbitdaByYear(acquisition, targetYear) { // acquisition = { ebitda: 500, completionQuarter: "Q3 26" } const dealYear = extractYear(acquisition.completionQuarter); return (dealYear === targetYear) ? acquisition.ebitda : 0; } // For each financial year, sum all acquisitions completing that year: function getYearAcquisitionEbitda(acquisitions, targetYear) { return acquisitions.reduce((sum, acq) => { return sum + routeEbitdaByYear(acq, targetYear); }, 0); }
Key for HTML app: The online tool should let users set each acquisition's quarter from a dropdown (Q1–Q4) and year. The routing function then determines which annual (or quarterly) column each deal's financials appear in. This replaces the Excel IF-based column routing with a simple programmatic filter.

3Core Business Revenue

Survey Solutions has three operating entities (LSS, JWG, S/Tec) each with their own revenue line, growth rate, and margin structure. The core business forms the baseline before any acquisitions.

Gross Revenue Growth

Excel nuance: FY25 and FY26 values are hardcoded in the Excel model (not formula-driven). The growth formula =G73*(1+$L$73) only kicks in from FY27 onwards. The HTML app should use formula-driven growth from a base year, but be aware that Excel "actuals" for FY25/FY26 may differ from pure compound growth.
Excel // Row 73: Core entity gross revenue // LSS (row 73): F73 = 15924 // FY25 actual (hardcoded) G73 = 17310 // FY26 budget (hardcoded — NOT 15924 × 1.09) H73 = G73 * (1 + $L$73) // FY27 = 17310 × 1.09 = 18867.9 // JWG (row 74): F74 = 1802 // FY25 actual G74 = 1880 // FY26 budget (hardcoded) H74 = G74 * (1 + $L$74) // Growth at 5% // S/Tec (row 75): F75 = 1495 (= 1007 + 488) // FY25 actual (two sub-entities) G75 = 1480 // FY26 budget (hardcoded, slight decline) H75 = G75 * (1 + $L$75) // Growth at 5% // Row 76: Total core gross revenue F76 = SUM(F73:F75) // Sum of 3 entities
JavaScript // ACTUAL values from Excel (hardcoded for FY25 and FY26) const coreEntities = [ { name: "LSS", fy25: 15924, fy26: 17310, growthRate: 0.09 }, { name: "JWG", fy25: 1802, fy26: 1880, growthRate: 0.05 }, { name: "S/Tec", fy25: 1495, fy26: 1480, growthRate: 0.05 }, ]; function calcCoreGrossRevenue(entity, yearIndex) { // yearIndex: 0 = FY25, 1 = FY26, 2 = FY27, etc. if (yearIndex === 0) return entity.fy25; if (yearIndex === 1) return entity.fy26; // FY27+ uses compound growth from FY26 base return entity.fy26 * Math.pow(1 + entity.growthRate, yearIndex - 1); } function totalCoreGrossRevenue(yearIndex) { return coreEntities.reduce((sum, e) => sum + calcCoreGrossRevenue(e, yearIndex), 0); }

Net Revenue (Subcontractor Deduction)

Excel nuance: The subcontractor retention rate varies by year: 90% (FY25), 91.446% (FY26), 91% (FY27+). This likely reflects an improving subcon mix as the business scales.
Excel F77 = F76 * 0.9 // FY25: Net Revenue = 90% of Gross G77 = G76 * 0.91446 // FY26: 91.446% retention H77 = H76 * 0.91 // FY27: 91% retention I77 = I76 * 0.91 // FY28: 91% retention
JavaScript const SUBCON_RETENTION = { 0: 0.90, // FY25 1: 0.91446, // FY26 2: 0.91, // FY27+ }; function getSubconRetention(yearIndex) { return SUBCON_RETENTION[yearIndex] ?? 0.91; // Default to 91% for FY27+ } function coreNetRevenue(yearIndex) { return totalCoreGrossRevenue(yearIndex) * getSubconRetention(yearIndex); }

Gross Profit

Excel // Row 80: GP = margin × net revenue G80 = G81 * G77 // GP margin × net revenue // G81 holds the margin: 42.9% (FY26) → 43.0% (FY27+)

Overheads

Excel // Row 83: Overheads = core gross revenue × OH% × -1 F83 = F76 * F84 * -1 // Negative value (cost) // F84 = 22.4% (FY25), 22.5% (FY26 onwards)

Core EBITDA

Excel F87 = F80 + F83 // Core EBITDA = Gross Profit + Overheads (overheads are negative)
JavaScript function calcCoreEbitda(yearIndex) { const grossRev = totalCoreGrossRevenue(yearIndex); const netRev = grossRev * SUBCON_RETENTION; const gpMargin = yearIndex === 0 ? 0.429 : 0.43; const ohPct = yearIndex === 0 ? 0.224 : 0.225; const grossProfit = netRev * gpMargin; const overheads = grossRev * ohPct * -1; // Note: OH% applied to GROSS revenue, not net return grossProfit + overheads; // overheads already negative }
Important detail: Overheads are calculated as a percentage of gross revenue (row 76), while gross profit is calculated on net revenue (row 77). This is a deliberate distinction in the Excel model — overheads scale with total business activity including subcontractors, while profit margins apply only to retained revenue.

4Acquisition Revenue

Each acquisition's revenue is back-calculated from EBITDA using an assumed margin. The Excel model does not track acquisition revenue independently — it derives it from EBITDA and a blended margin.

Revenue Back-Calculation

Excel // Row 38 onwards: "Acquisitions Revenue by Year of Completion" G38 = G48 / $H$29 // Revenue = EBITDA / Margin // $H$29 is the EBITDA margin for the first pipeline deal // Margin is typically ~25-30%
JavaScript function acquisitionRevenue(ebitda, ebitdaMargin) { // ebitdaMargin from the deal (e.g., 0.27 = 27%) return ebitda / ebitdaMargin; }
Note: The Excel model uses a single cell reference ($H$29) for the margin of all pipeline acquisitions. In the HTML app, each acquisition should carry its own EBITDA margin input so revenue can be computed per deal.

5Group Revenue Build-Up

Group revenue combines the core business, pipeline acquisitions (pre-existing, with known financials), and new bolt-on acquisitions. The formula is complex because it handles margin-based revenue conversion for acquired businesses.

The Group Revenue Formula

Excel // Row 90: Group Revenue G90 = (G14 + G46) // Pipeline acquisition revenue (direct) + IF(F101 > 0, (G94 / F101), 0) // Acquired EBITDA → revenue via implied margin + G76 // Core business gross revenue // F101: Implied margin for acquired businesses F101 = IFERROR(F94 / (F14 + F46), 0) // Total acquired EBITDA / total pipeline revenue

The key insight: for acquired businesses, the model uses an implied EBITDA margin derived from the prior year's ratio of acquired EBITDA to pipeline revenue. This converts EBITDA back to revenue for the group P&L.

JavaScript function calcGroupRevenue(year) { const coreRev = totalCoreGrossRevenue(year.index); // Pipeline acquisitions (pre-existing, with known revenue) const pipelineRev = year.pipelineRevenue; // Direct revenue figures // Bolt-on acquisitions: convert EBITDA to revenue using implied margin const totalAcqEbitda = year.totalAcquiredEbitda; const prevImpliedMargin = year.prev ? year.prev.impliedMargin : 0; const acqRevenue = prevImpliedMargin > 0 ? totalAcqEbitda / prevImpliedMargin : 0; return coreRev + pipelineRev + acqRevenue; } function calcImpliedMargin(year) { // Implied margin = total acquired EBITDA / total pipeline revenue const pipelineRev = year.pipelineRevenue; if (pipelineRev === 0) return 0; return year.totalAcquiredEbitda / pipelineRev; }
HTML app simplification: Rather than this circular implied-margin approach, the HTML app should store each acquisition's EBITDA margin explicitly, then compute revenue = EBITDA / margin directly. This eliminates the IFERROR guards and cross-year margin lookups.

JavaScript const ACQ_GROWTH_RATE = 0.05; // 5% annual growth on acquired EBITDA function calcVintageBaseEbitda(acquisitions, pipelineDeals, targetYear) { // Sum pipeline EBITDA + synergies routed to this year let total = 0; // Pipeline deals (pre-agreed, hardcoded in the model) for (const deal of pipelineDeals) { total += deal.ebitdaByYear[targetYear] || 0; total += deal.synergyByYear[targetYear] || 0; } // Bolt-on acquisitions completing this year for (const acq of acquisitions) { total += routeEbitdaByYear(acq, targetYear); } // Synergies for ALL acquisitions that phase into this year total += totalSynergiesForYear(acquisitions, targetYear); return total; } function calcVintageEbitda(vintageBase, yearsSinceCompletion) { if (yearsSinceCompletion === 0) return vintageBase; return vintageBase * Math.pow(1 + ACQ_GROWTH_RATE, yearsSinceCompletion); }

Group EBITDA (All Layers Combined)

Excel // Row 98: Group EBITDA = sum of all layers F98 = SUM(F93:F97) // Where: // F93 = Core EBITDA (row 87, after adding central costs/exceptionals) // F94 = Acquired EBITDA (current year vintage) // F95 = Prior year 1 vintage, grown // F96 = Prior year 2 vintage, grown // F97 = Additional/central adjustments
JavaScript function calcGroupEbitda(yearIndex, allVintages) { const coreEbitda = calcCoreEbitda(yearIndex); // Sum all vintage layers (each growing at 5% from their base year) let acquiredEbitda = 0; for (const vintage of allVintages) { const yearsSince = yearIndex - vintage.completionYearIndex; if (yearsSince >= 0) { acquiredEbitda += calcVintageEbitda(vintage, yearsSince); } } return coreEbitda + acquiredEbitda; }

7Synergy Phase-In Model

Synergies don't arrive immediately — they phase in over Year 1 and Year 2 based on which quarter the deal completes. A Q1 deal gets 75% of synergies in Year 1 and the remaining 25% in Year 2. A Q4 deal gets 0% in Year 1 and 100% in Year 2.

Phase-In Calculation

Excel // Lookup table at AB58:AD61 (Quarter → Year1 pct, Year2 pct) // AB58="Q1", AC58=0.75, AD58=0.25 // AB59="Q2", AC59=0.50, AD59=0.50 // AB60="Q3", AC60=0.25, AD60=0.75 // AB61="Q4", AC61=0.00, AD61=1.00 // For Year 1 (same year as completion): U58 = IF(RIGHT($S58, 2) = U$57, 1, 0) // Does deal complete this year? (1 or 0) * VLOOKUP(LEFT($S58, 2), $AB$58:$AC$61, 2, 0) // Lookup Year1 phase-in % // For Year 2 (year after completion) — TWO terms added: X58 = IF(RIGHT($S58, 2) = X$57, 1, 0) // Does deal complete THIS year? * VLOOKUP(LEFT($S58, 2), $AB$58:$AC$61, 2, 0) // Year1 phase-in for new deals + IF(RIGHT($S58, 2) = W$57, 1, 0) // Did deal complete LAST year? * VLOOKUP(LEFT($S58, 2), $AB$58:$AD$61, 3, 0) // Year2 phase-in for prior year deals // Synergy value = Total synergy × phase-in fraction F58 = $L58 * U58 // £ synergy in this year
JavaScript const SYNERGY_PHASEIN = { "Q1": { year1: 0.75, year2: 0.25 }, "Q2": { year1: 0.50, year2: 0.50 }, "Q3": { year1: 0.25, year2: 0.75 }, "Q4": { year1: 0.00, year2: 1.00 }, }; function calcSynergyForYear(acquisition, targetYear) { const dealYear = extractYear(acquisition.completionQuarter); const dealQ = extractQuarter(acquisition.completionQuarter); // "Q3" const phaseIn = SYNERGY_PHASEIN[dealQ]; const yearDiff = parseInt(targetYear) - parseInt(dealYear); if (yearDiff === 0) { // Year of completion → Year 1 phase-in return acquisition.totalSynergy * phaseIn.year1; } else if (yearDiff === 1) { // Year after completion → Year 2 phase-in (remaining) return acquisition.totalSynergy * phaseIn.year2; } else if (yearDiff > 1) { // Fully synergised — no additional phase-in (growth handled separately) return 0; // Already in the base via vintage growth } return 0; // Before completion } // Total synergies for a given year across all acquisitions function totalSynergiesForYear(acquisitions, targetYear) { return acquisitions.reduce((sum, acq) => { return sum + calcSynergyForYear(acq, targetYear); }, 0); }
HTML app enhancement: For a quarterly model, you could extend this to quarter-level granularity. A Q3 deal would get: Q3 synergies = 0%, Q4 = 25% annualised, then Year 2 Q1-Q4 = remaining 75%. The current Excel model only does annual phase-in, but quarterly allows more precise cash flow timing.

9Capex & HP (Hire Purchase) Financing

Capital expenditure is split 50/50 between cash capex and HP-financed equipment. HP has its own interest and capital repayment schedules that feed into the cash flow waterfall.

Total Capex

Excel F107 = -F90 * $L$107 // Total capex = 3% of group revenue (negative) // L107 = 0.03 (3%)

Cash Capex (50%)

Excel F113 = F107 / 2 // Half of total capex paid in cash

HP Capital Repayment

Excel F112 = F107 / 2 / 4 // Current year HP capex repaid over 4 years + ($E$107 / 4) // Legacy HP balance also repaid over 4 years // In subsequent years, this becomes cumulative: G112 = G107 / 2 / 4 // New HP this year + F112 // Plus carry-forward of prior HP repayments

HP Interest

Excel F111 = ((F107 / 2) * $L$111) // Interest on current year HP amount + (F107 / 2 / 4) // Capital element + (E107 * L111) // Interest on legacy HP balance // L111 = 8.5%
JavaScript const CAPEX_PCT = 0.03; const HP_SPLIT = 0.5; // 50% of capex is HP-financed const HP_TERM = 4; // 4-year repayment const HP_RATE = 0.085; // 8.5% HP interest rate function calcCapex(groupRevenue) { const totalCapex = groupRevenue * CAPEX_PCT; const cashCapex = totalCapex * HP_SPLIT; const hpAmount = totalCapex * HP_SPLIT; return { totalCapex, cashCapex, hpAmount }; } function calcHpSchedule(years) { let outstandingHp = 0; // Legacy HP balance from E107 return years.map((year, i) => { const { hpAmount } = calcCapex(year.groupRevenue); // Capital repayment = new HP / 4 years + prior outstanding / 4 years const capitalRepayment = (hpAmount / HP_TERM) + (outstandingHp / HP_TERM); // Interest = rate × (new HP + outstanding) const hpInterest = HP_RATE * (hpAmount + outstandingHp); // Update outstanding balance outstandingHp = outstandingHp - (outstandingHp / HP_TERM) + hpAmount; return { hpCapital: -capitalRepayment, // negative (cash out) hpInterest: -hpInterest, // negative (cash out) cashCapex: -(hpAmount), // negative (cash out) }; }); }
Excel model nuance: The HP calculation in the Excel model has a slightly different structure than a clean amortisation — it uses a chain formula where each year's capital repayment includes the carry-forward from prior years. The HTML app should implement a proper HP amortisation table tracking opening balance, additions, repayments, and closing balance per year (and per quarter in the quarterly model).

10Working Capital

Working capital movement is modelled simply as 2 months of EBITDA growth — when EBITDA grows, cash is absorbed into working capital; when it shrinks, cash is released.

Excel F114 = -(F98 - E98) / 6 // WC = -(current EBITDA - prior EBITDA) / 6 // Dividing annual growth by 6 = 2 months' worth // Negative because growth absorbs cash
JavaScript function calcWorkingCapital(currentEbitda, priorEbitda) { // 2 months of EBITDA growth = annual growth / 6 return -(currentEbitda - priorEbitda) / 6; }
Quarterly conversion: For a quarterly model, working capital should be: -(Q_ebitda - prev_Q_ebitda) / 6. Since the quarter already represents ~3 months, dividing by 6 gives roughly 2 weeks of quarterly growth. Alternatively, consider using ebitda_growth × (2/12) to keep the 2-month logic but apply it quarterly: -(quarterly_ebitda_annualised - prev) × (2/12) / 4.

11Interest Calculations

Bank Debt Interest

Excel G110 = ((G192 * $L$110) + (G193 * L110)) * -1 // G192 = Opening bank debt balance // G193 = Drawdown during the year // L110 = 10.75% annual interest rate // Interest charged on BOTH opening balance AND new drawdowns
JavaScript const BANK_RATE = 0.1075; // 10.75% function calcBankInterest(openingDebt, drawdown) { // Interest on opening balance + interest on new drawdown (full year) return -((openingDebt * BANK_RATE) + (drawdown * BANK_RATE)); }
Quarterly improvement: The annual model charges a full year's interest on drawdowns regardless of when they occur. In a quarterly model, a Q3 drawdown would only incur ~50% of annual interest in its first year (2 quarters). Formula: drawdown × rate × (quartersRemaining / 4).

12Tax Treatment

Excel F115 = 0 // Tax = £0 across entire model period // Due to historic losses brought forward
JavaScript // For the current model: tax = 0 due to losses brought forward // Future enhancement: track taxable profit and loss pool function calcTax(taxableProfit, lossesBroughtForward) { const netTaxable = Math.max(0, taxableProfit - lossesBroughtForward); return -(netTaxable * 0.25); // UK corporation tax rate }

13Funding Requirement per Acquisition

Each acquisition has a total funding requirement = EBITDA × purchase multiple + advisory fees.

Excel // Row 128: Total funding required per deal G128 = IF(G48 > 0, (G48 * $L48 + $M48), 0) // G48 = EBITDA routed to this year // L48 = Purchase multiple (e.g., 5.0x) // M48 = Fees (e.g., £75k) // Only calculated if deal completes in this year
JavaScript function calcFundingRequired(acquisition) { return (acquisition.ebitda * acquisition.purchaseMultiple) + acquisition.fees; } // Total funding for a year = sum of all deals completing that year function totalYearFunding(acquisitions, targetYear) { return acquisitions .filter(a => extractYear(a.completionQuarter) === targetYear) .reduce((sum, a) => sum + calcFundingRequired(a), 0); }

14Leverage Covenant Test

Before each acquisition, the model tests whether the resulting leverage (debt / EBITDA) would breach the bank's covenant. The covenant ratio steps up as group EBITDA grows, allowing more leverage for larger platforms.

Running EBITDA (for covenant test)

Excel // L29: Running total EBITDA including core business + all acquisitions to date L29 = (IF(M29="26", F$87, IF(M29="27", G$87, H$87))) // Core EBITDA for that year + SUM($G$29:G29) // Cumulative acquired EBITDA

Covenant Lookup

Excel // Covenant thresholds (rows 220-223): // EBITDA < £3,500 → 3.00x leverage // EBITDA £3,500–5,000 → 3.25x leverage // EBITDA > £5,000 → 3.75x leverage // EBITDA > £10,000 → 4.00x (not yet activated) N29 = IF($L29 < $E$220, $F$220, // < £3.5m → 3.0x IF(AND($L29 > $E$220, $L29 < $E$222), $F$221, // £3.5-5m → 3.25x IF($L29 > $E$222, $F$222, 0))) // > £5m → 3.75x
JavaScript const COVENANTS = [ { threshold: 3500, multiple: 3.00 }, // Below £3.5m EBITDA { threshold: 5000, multiple: 3.25 }, // £3.5m – £5m { threshold: 10000, multiple: 3.75 }, // £5m – £10m { threshold: Infinity, multiple: 4.00 }, // Above £10m ]; function getCovenantMultiple(runningEbitda) { for (const covenant of COVENANTS) { if (runningEbitda < covenant.threshold) { return covenant.multiple; } } return COVENANTS[COVENANTS.length - 1].multiple; } function calcRunningEbitda(acquisition, coreEbitda, priorAcqEbitda) { // Core EBITDA for the year + cumulative acquired EBITDA up to this deal return coreEbitda + priorAcqEbitda + acquisition.ebitda; }
Critical ordering: Acquisitions must be processed sequentially within each year because each deal's covenant test depends on the cumulative EBITDA including all prior deals. The HTML app should process acquisitions in order (Q1 before Q2 before Q3 before Q4 within each year).

15Debt Capacity per Acquisition

Debt capacity = the covenant multiple × deal EBITDA + fees. This is the maximum bank debt available for each deal.

Excel // Row 143: Debt capacity per deal G143 = IF(G48 > 0, (N29 * (G48) + $M48), 0) // N29 = Covenant multiple (from leverage test) // G48 = Deal EBITDA // M48 = Fees // For pre-existing pipeline deals (different formula): F139 = IF(F16 > 0, (F$186 * (F16 + F22) + $M16), 0) // F$186 = Year-level leverage multiple (from debt schedule section)
JavaScript function calcDebtCapacity(acquisition, covenantMultiple) { return (covenantMultiple * acquisition.ebitda) + acquisition.fees; }

16Deferred Consideration

Part of the purchase price is deferred — typically 25% deferred by 1 year, with larger deals also having a 15% 2-year deferral. This reduces the upfront funding requirement.

Excel // Row 160: 1-year deferred consideration G160 = G48 * ($L48 * $L$159) // EBITDA × multiple × 25% // L159 = 0.25 (25% deferred 1 year) // Row 165: 2-year deferred consideration (larger FY28 deals only) I165 = I53 * ($L53 * $L$159) // 25% 1-year deferral + I53 * ($L53 * $L$160) // + 15% 2-year deferral // L160 = 0.15 (15% deferred 2 years)
JavaScript const DEFERRAL_1YR = 0.25; // 25% deferred 1 year const DEFERRAL_2YR = 0.15; // 15% deferred 2 years (larger deals) function calcDeferrals(acquisition) { const purchasePrice = acquisition.ebitda * acquisition.purchaseMultiple; const defer1yr = purchasePrice * DEFERRAL_1YR; // 2-year deferral only for deals above a size threshold const defer2yr = acquisition.hasTwoYearDeferral ? purchasePrice * DEFERRAL_2YR : 0; return { totalDeferred: defer1yr + defer2yr, defer1yr, defer2yr, dueYear1: defer1yr, // Cash outflow in year +1 dueYear2: defer2yr, // Cash outflow in year +2 }; }
Cash flow timing: Deferred consideration reduces year-0 funding but creates cash obligations in future years. The HTML app must track when deferred amounts fall due and include them in the appropriate quarter/year's cash outflows. This is essential for accurate cash flow forecasting.

17Equity Requirement

Equity is the residual — whatever can't be funded by bank debt or deferrals must come from investor equity.

Excel // Row 175: Equity required per deal G175 = IF(G128 - G143 - G160 > 0, (G128 - G143 - G160), 0) // G128 = Total funding required // G143 = Debt capacity // G160 = Deferred consideration // If debt + deferrals cover everything, equity = 0

18Debt Schedule

The debt schedule tracks the bank facility — opening balance, drawdowns per acquisition, and available headroom. Drawdowns are constrained by the total facility capacity.

Total Facility Capacity

Excel // Row 185: Group EBITDA (reference to row 98) G185 = G98 // Row 186: Leverage multiple for the year // = maximum covenant multiple of all deals completing that year G186 = MAXIFS(N29:N36, M29:M36, "26") // Takes the highest covenant ratio from all '26 deals // Row 187: Total debt capacity G187 = G186 * G185 // Leverage × Group EBITDA

Drawdown Mechanics

Excel // Row 189: Available headroom G189 = IF(G187 - G192 > 0, G187 - G192, 0) // G187 = Total capacity, G192 = Opening balance // Row 192: Opening balance = prior year closing G192 = F194 // Row 193: Drawdown during year G193 = IF(G189 > 0, MIN((G151 + F168), G189), 0) // G151 = debt capacity for this year's deals // F168 = deferred consideration from prior year falling due // Capped at available headroom // Row 194: Closing balance G194 = SUM(G192:G193) // Opening + Drawdown // Row 195: Remaining headroom G195 = IF(G187 - G194 > 0, G187 - G194, 0)
JavaScript function buildDebtSchedule(years) { let openingBalance = 0; return years.map(year => { // Total capacity = leverage multiple × group EBITDA const maxCovenant = Math.max(...year.deals.map(d => d.covenantMultiple)); const totalCapacity = maxCovenant * year.groupEbitda; // Available headroom const headroom = Math.max(0, totalCapacity - openingBalance); // Required drawdown = new deal debt + prior deferred falling due const requiredDrawdown = year.newDealDebt + year.deferredFallingDue; // Actual drawdown (capped at headroom) const drawdown = Math.min(requiredDrawdown, headroom); // Closing balance const closingBalance = openingBalance + drawdown; const remainingHeadroom = Math.max(0, totalCapacity - closingBalance); const result = { openingBalance, drawdown, closingBalance, totalCapacity, headroom, remainingHeadroom, }; openingBalance = closingBalance; // Carry forward return result; }); }

19WestBridge Loan Note (Compounding Interest)

WestBridge Capital provides a loan note at the platform level that compounds annually at 12.5%. Interest is not paid in cash — it accrues and compounds, growing the total obligation over time.

Excel // X17 = Principal (loan note amount) // V18 = 12.5% annual rate // Row 210: Accrued interest per year E210 = $X$17 * $V$18 // Year 1: principal × 12.5% F210 = ($X$17 + E210) * $V$18 // Year 2: (principal + yr1 interest) × 12.5% G210 = ($X$17 + E210 + F210) * $V$18 // Year 3: compound H210 = ($X$17 + E210 + F210 + G210) * $V$18 // Year 4: compound I210 = ($X$17 + SUM(E210:H210)) * $V$18 // Year 5: compound
JavaScript const WB_RATE = 0.125; // 12.5% annual compounding function calcWestBridgeSchedule(principal, years) { let totalAccrued = 0; const schedule = []; for (let i = 0; i < years; i++) { const yearInterest = (principal + totalAccrued) * WB_RATE; totalAccrued += yearInterest; schedule.push({ year: i + 1, openingBalance: principal + totalAccrued - yearInterest, interestAccrued: yearInterest, closingBalance: principal + totalAccrued, }); } return { schedule, totalOwed: principal + totalAccrued, // At exit totalInterest: totalAccrued, }; }
Key: WestBridge interest does NOT appear in the cash flow waterfall — it only crystalises at exit when the loan note is repaid from exit proceeds. This means the cash flow model is "cleaner" but the exit waterfall must deduct the full compounded amount.

23Calculation Order (Dependency Graph)

The calculations have strict dependencies. The HTML app must execute them in this order to avoid circular references.

Execution Order STEP 1: Parse inputs ├── Core entities (base revenue, growth rates, margins) ├── Acquisition list (from shortlist, with quarters assigned) └── Model parameters STEP 2: For each period (year or quarter), in chronological order: 2a. Core Business ├── Core gross revenue = base × (1 + growth)^n ├── Core net revenue = gross × subcon retention ├── Core GP = net revenue × GP margin ├── Core overheads = gross revenue × OH% × -1 └── Core EBITDA = GP + overheads 2b. Acquisition Routing ├── Route each deal's EBITDA to completion period ├── Calculate synergy phase-in amounts └── Pro-rate for partial periods (quarterly model) 2c. EBITDA Build-Up ├── New acquisition EBITDA (deals completing this period) ├── Synergies phasing in ├── Prior vintage EBITDA (grown at 5%) └── Group EBITDA = core + all acquired layers 2d. Leverage & Funding (per acquisition, in sequence) ├── Running EBITDA (core + cumulative acquired) ├── Covenant multiple lookup ├── Funding required = EBITDA × multiple + fees ├── Debt capacity = covenant × EBITDA + fees ├── Deferred consideration └── Equity = MAX(0, funding - debt - deferred) 2e. Revenue (depends on EBITDA + margins) ├── Acquired revenue = acquired EBITDA / margin └── Group revenue = core + pipeline + acquired 2f. Cash Flow Waterfall ├── Capex = 3% of group revenue → split cash / HP ├── HP schedule (interest + capital repayment) ├── Working capital = 2 months of EBITDA growth ├── Bank interest = (opening debt + drawdown) × rate ├── Tax = 0 (losses b/f) ├── Net cash = EBITDA + all the above └── Cumulative cash = prior + net cash 2g. Debt Schedule ├── Opening = prior closing ├── Capacity = leverage × group EBITDA ├── Headroom = capacity - opening ├── Drawdown = MIN(required, headroom) └── Closing = opening + drawdown STEP 3: Exit & Returns (once, at end of projection) ├── WestBridge compound interest schedule ├── Exit EV = EBITDA × exit multiple ├── Equity waterfall (EV + cash - debt - notes - HP - deferrals) ├── Investor splits └── MoM return
Note on circular references: The Excel model has a near-circular dependency: bank interest depends on debt drawdown, which depends on debt capacity, which depends on EBITDA, which drives the cash flow that affects cumulative cash. However, this is resolved because bank interest uses opening balance + drawdown (not closing), and debt capacity is based on EBITDA (not cash). The HTML app should follow the same sequential approach — never reference a period's closing debt balance in that same period's interest calculation.

BAppendix: Excel Model Quirks & Notes

1. Hardcoded FY25/FY26 values: Core entity revenues (rows 73–75) are hardcoded for FY25 and FY26. The compound growth formula only applies from FY27 onwards. The HTML app should allow users to override any year's value or let it auto-calculate.
2. Variable subcontractor retention: Net revenue uses different multipliers each year — 90% (FY25), 91.446% (FY26), 91% (FY27+). This should be a configurable parameter per year in the HTML app.
3. FY28 leverage formula bug: Cell I186 uses MAXIFS(N29:N36, M29:M36, 27) instead of 27 → 28. This means FY28's facility leverage uses FY27's covenant multiple (3.75x) instead of potentially looking up a higher FY28 value. The HTML app should use the correct year.
4. Two WestBridge loan notes: The model tracks TWO separate loan note tranches:
• Tranche 1 (row 210): Principal at X17 = -£15,721k, rate at V18 = 12.5%
• Tranche 2 (row 212): Principal at X19, rate at V20
Both compound independently and sum at row 214.
5. Acquisition synergies for deals 1 & 2: These have zero synergies (L58=0, L59=0) in the current model, even though they are the first bolt-on deals. This may be intentional (pure EBITDA consolidation) or an area for the user to adjust.
6. Column L dual-purpose: In the acquisition rows (29–36), column L contains the running total EBITDA (for covenant testing), NOT the purchase multiple. The purchase multiples are in L48:L55 (the EBITDA-by-year section). The HTML app should keep these as separate data fields.
